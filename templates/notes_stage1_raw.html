// CONCEPT END // // CONCEPT // HTML lingo

HTML language follows a pretty strict syntax and there are many concepts to learn to be able to write funtional HTML as well as being able to read the guidelines for the language. There are tags (and some require closing brackets, but others don't), inline tags, block tags, void tags, elements, attributes, values and probably a lot more stuff to keep track of.

// CONCEPT END // // CONCEPT // HTML structure

All HTML documents that are meant to display web content and be compatible with common browsers have to follow a set syntax. All such HTML documents share a tree-like backbone made up of tags for doctype, html, head, meta-data, (css, javascript) and body (where the main contents of the page goes). It looks like this in a nutshell: 

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Web page title</title>
</head>
<body>
</body>
</html>
// CONCEPT END // // CONCEPT // Images, inline vs block

There are special tags for showing images denoted with a void tag: <img src='smiley.jpg' alt='Smiley face'>. A void tag means there is no closing tag (e.g. </img> for these tag types. It's confusing but that's just one of those things. Image elements in img tags appear inline as opposed to block elements that take up a complete row on the page by themselves (e.g. the <div> element).

Examples of Block Elements: <p>, <div>, <form>, <header>, <nav>, <ul>, <li>, and <h1>.

Examples of Inline Elements: <a>, <span>, <b>, <em>, <i>, <cite>, <mark>, and <code>.

// CONCEPT END // // LESSON // Lesson 1.2: Creating a Structured Document // CONCEPT // Structure, structure, structure

HTML is the language that provides the backbone of the webpage's content in a branched order - the so-called DOM or Document Object Model. This clever approach makes it highly adaptable in a controlled way by applying formatting to the backbone elements separately - namely via CSS. It enables me to apply a complete "style make-over" to collective parts of a webpage with minimum effort and time use.

// CONCEPT END // // CONCEPT // Styling

On such a "style make-over" the outputted webpage looks different in the browser. The HTML (structure) remains unchanged while the CSS (style) changes and thus controls how the elements of the page appear on screen.

// CONCEPT END // // CONCEPT // Boxification

It appears that all elements in HTML is contained in rectangular boxes - their dimensions and other parameters may be defined via separate styles. This is another clever part of HTML as it enables easy page adaptation to various screen sizes, browsers and digital platforms in general.
Boxes are created in HTML with block element generating tags like the <div>-tag (but not around the <span>-tag). Creating a box inside another box can be done by enclosing one <div>-tag inside another. Clever! So when I break down a webpage as it looks in the browser and "read" the invisible "boxes", I can actually mimic those boxes' layout into my own HTML by building a hierarchy of <div>-tags.

The css styling enables me to manipulate how my boxes and their contents will appear on screen. I may want to show the borders of the boxes or I can choose not to and keep them invisible. I may want to add somespace around my content inside my box or maybe add space around my boxes so they appear further from each other. Styling by CSS is linked to HTML elements by labeling the HTML elements (tags) with class and id attributes. This seems to be the stuff that makes the magic happen.

// CONCEPT END // // CONCEPT // Cascading and inheritance

HTML structure is DOM, and CSS is applied cascading throughout the DOM in turn dictating the order of the applied styling. The CSS for my HTML page can be "called" or referenced into my HTML from a separately saved CSS file. It can also be written directly into the beginning of my HTML. I guess the latter is great for smaller projects while the former seems very useful for big projects with tons of HTML files all calling the same external CSS file.

CSS applied to an element is inherited to the descendants of that element, but there are differences in how e.g. text- and box-properties are inherited. This seems awesome and it all gets more logical and useful at this stage. There's just the thing about remembering all of this...

// CONCEPT END // // CONCEPT // Repetition

Programming seems a lot about avoiding unnecessary repetition. CSS together with the inherent structures of HTML and CSS are very efficient at that. E.g. make one CSS file and call this from a bunch of HTML files to easy apply a uniform layout to all pages in no time. The rules of inheritance in CSS also helps programmers write less code because they can simply apply style to a higher-level or ancestor-level element.
